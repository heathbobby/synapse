## Purpose

Define implementation-level constraints, interfaces, data contracts, operational
assumptions, and technology-neutral specifications for Synapse. These
specifications intentionally avoid unsupported stack commitments and should be
refined as requirements, MVP scope, and compliance constraints stabilize.

## System Capabilities and Component Specifications

### Visual Workflow Designer

**Purpose**: Let human operators, SMEs, and workflow owners author, compose,
monitor, and revise domain-agnostic agent workflows.

**Required capabilities**

- Canvas-based workflow authoring with steps, dependencies, role assignments,
  approval gates, and integration boundaries.
- Nested workflow composition so repeatable expert processes can be reused
  inside larger processes.
- Live monitor views for workflow run state, blocked steps, pending approvals,
  agent completion signals, and exception telemetry.
- Versioned workflow templates with publish, retire, and rollback semantics.
- Source links from workflow steps to canonical requirements, standards,
  knowledge assets, or decision records.

**Open implementation choices**

- UI framework, graph representation, template serialization format, and
  workflow diffing strategy.

### Workflow Orchestration Runtime

**Purpose**: Instantiate workflow templates, coordinate humans and agents, track
state, and emit lifecycle events.

**Required capabilities**

- Execute directed workflow graphs with explicit dependencies.
- Support pause, resume, retry, branch, cancel, and approval-required states.
- Dispatch work to agents or human reviewers with complete canonical context.
- Preserve run history, state transitions, artifacts, and completion signals.
- Enforce workflow-level completion criteria through deterministic checks where
  possible.
- Emit state changes to the hybrid event bus.

**Runtime states to support**

- `draft`
- `ready`
- `running`
- `waiting_for_agent`
- `waiting_for_human_approval`
- `blocked`
- `completed`
- `failed`
- `cancelled`

### Persona Registry and OOP Management

**Purpose**: Manage reusable agent coworker capabilities using an
inheritance-like persona model.

**Required capabilities**

- Store base persona templates, extended persona templates, and task-specific
  persona instances.
- Resolve persona lineage at task dispatch time.
- Track persona version, owner, intended domain, allowed tools, evidence
  requirements, quality standards, and completion protocol.
- Support review and promotion of persona changes generated by the knowledge
  loop.
- Prevent hidden prompt drift by requiring persona changes to be versioned and
  attributable.

**Conceptual structure**

```text
BasePersona
  -> ExtendedPersona
       -> PersonaInstance
```

### Agent Task Dispatcher

**Purpose**: Convert workflow steps into executable agent tasks.

**Required capabilities**

- Generate task cards/prompts from workflow step, persona, canonical context,
  deliverables, and standards.
- Include completion signals aligned with `docs/standards/AI_AGENT_STANDARDS.md`.
- Track assigned agent, task status, artifact paths, source references, and
  telemetry.
- Support safe parallel dispatch only when tasks write to disjoint deliverables
  or otherwise have explicit coordination contracts.
- Route partial completions and token-budget issues into recovery workflows or
  follow-up backlog items.

### Knowledge and SME Grounding Layer

**Purpose**: Ground agent work in canonical truth and codified tribal knowledge.

**Required capabilities**

- Index canonical requirements, architecture, standards, planning docs, work
  items, runbooks, approved extracts, and SME-curated knowledge assets.
- Record evidence class for claims: source-backed, inferred, assumed, or open.
- Link knowledge assets to source documents, human reviewers, confidence levels,
  and freshness/expiration metadata.
- Support retrieval by workflow, domain, persona, work item, incident, and
  integration context.
- Route reusable learnings into canonical docs, workflow templates, or personas
  after review.

### Hybrid Event Bus

**Purpose**: Carry asynchronous telemetry and integration events while also
supporting synchronous human-in-the-loop approval semantics.

**Required capabilities**

- Publish/subscribe lifecycle events for workflows, tasks, personas, knowledge
  updates, approvals, integrations, and telemetry.
- Request/response or correlation support for approval gates.
- Schema versioning, event ownership, idempotency keys, retry behavior,
  dead-letter handling, and trace correlation.
- Event replay or audit reconstruction capability for operational review.
- Clear contract boundaries between core platform events and domain-specific
  extension events.

### Human Approval Coordinator

**Purpose**: Make review and accountability explicit in agentic workflows.

**Required capabilities**

- Define approval gates in workflow templates.
- Present decision context: generated artifact, source evidence, risk labels,
  recommendation, affected downstream steps, and policy basis.
- Record approve, reject, request-changes, or escalate outcomes.
- Resume, branch, or halt workflow execution based on approval decision.
- Emit approval events and durable audit records.

### Legacy Bridge

**Purpose**: Connect Synapse to existing systems during transition-state
operations without coupling legacy specifics into the domain-agnostic core.

**Required capabilities**

- Adapter pattern for source repositories, documentation systems, ticketing,
  operational tooling, monolith APIs, and communication channels.
- Read and write boundaries per adapter, including authentication, rate limits,
  error handling, and audit requirements.
- Requirements and knowledge extraction workflows for legacy modernization.
- Stabilization workflows that allow agents to assist with existing systems
  while feeding greenfield architecture and backlog refinement.

### Audit, Telemetry, and Decision Ledger

**Purpose**: Provide traceability and measurable continuous improvement.

**Required capabilities**

- Store workflow run history, task status, event history, approval decisions,
  persona versions, knowledge updates, and architecture decisions.
- Correlate generated artifacts to workflow step, persona version, source
  evidence, event stream, and reviewer approval.
- Measure completion rates, cycle time, failure causes, approval queues, quality
  issues, and repeated learning patterns.
- Support reporting for refinement loops and operational review.

## Conceptual Data Contracts

These are logical contracts, not physical schema commitments.

### WorkflowTemplate

| Field | Description |
| --- | --- |
| `id` | Stable template identifier. |
| `version` | Version used for run reproducibility. |
| `name` | Human-readable workflow name. |
| `domain_scope` | Domain or cross-domain scope; may be generic. |
| `steps` | Ordered or graph-based step definitions. |
| `nested_workflows` | References to reusable workflow templates. |
| `role_bindings` | Required persona roles per step. |
| `approval_gates` | Human review requirements and policies. |
| `completion_criteria` | Deterministic and review-based completion checks. |
| `source_references` | Canonical docs or knowledge assets informing template. |

### WorkflowRun

| Field | Description |
| --- | --- |
| `id` | Runtime instance identifier. |
| `workflow_template_id` | Template source. |
| `workflow_template_version` | Exact template version executed. |
| `state` | Current runtime state. |
| `inputs` | User, system, or integration-provided context. |
| `step_history` | State transitions and outputs per step. |
| `assigned_tasks` | Human and agent tasks created by the run. |
| `events` | Correlated event identifiers. |
| `artifacts` | Produced or modified outputs. |

### PersonaDefinition

| Field | Description |
| --- | --- |
| `id` | Persona identifier. |
| `version` | Persona version. |
| `kind` | `base`, `extension`, or `instance`. |
| `parent_id` | Optional parent persona for OOP lineage. |
| `role_boundaries` | Responsibilities and prohibitions. |
| `allowed_tools` | Tools or integrations the persona may use. |
| `evidence_rules` | Source classification and citation requirements. |
| `completion_protocol` | Required completion or partial-completion signal. |
| `quality_gates` | Standards applied to outputs. |

### TaskCard

| Field | Description |
| --- | --- |
| `id` | Executable task identifier. |
| `workflow_run_id` | Parent workflow run. |
| `step_id` | Workflow step source. |
| `assignee_type` | `agent` or `human`. |
| `persona_version` | Persona used for agent work. |
| `canonical_context` | Required source-of-truth inputs. |
| `deliverables` | Expected output paths or objects. |
| `dependencies` | Blocking tasks, approvals, or events. |
| `status` | Runtime task status. |
| `completion_signal` | Final task signal. |

### KnowledgeAsset

| Field | Description |
| --- | --- |
| `id` | Knowledge record identifier. |
| `kind` | Requirement, decision, SME note, lesson, runbook, extract, etc. |
| `source_references` | Source documents, runs, or reviewers. |
| `evidence_class` | Source-backed, inferred, assumed, or open. |
| `confidence` | Confidence level or review status. |
| `freshness` | Last reviewed and expiration metadata. |
| `applicability` | Domains, workflows, personas, or systems where relevant. |

### ApprovalDecision

| Field | Description |
| --- | --- |
| `id` | Approval decision identifier. |
| `workflow_run_id` | Parent run. |
| `approval_gate_id` | Gate definition. |
| `approver` | Human or group accountable for decision. |
| `decision` | Approve, reject, request changes, or escalate. |
| `rationale` | Reason for decision. |
| `evidence_reviewed` | Artifacts and knowledge reviewed. |
| `timestamp` | Decision time. |
| `resulting_state` | Workflow state after decision. |

## Event Contract Families

All event contracts must follow `docs/standards/EVENT_CONTRACT_STANDARDS.md`:
owner, schema version, producers, consumers, payload, retries, and dead-letter
behavior must be documented before implementation.

| Event family | Examples | Notes |
| --- | --- | --- |
| Workflow lifecycle | `workflow.started`, `workflow.paused`, `workflow.completed`, `workflow.failed` | Core runtime events. |
| Workflow step | `workflow_step.started`, `workflow_step.completed`, `workflow_step.blocked` | Enables live monitoring and dependency handling. |
| Agent task | `agent_task.dispatched`, `agent_task.progressed`, `agent_task.completed`, `agent_task.partial` | Captures coworker execution. |
| Approval | `approval.requested`, `approval.decided`, `approval.expired`, `approval.escalated` | Must support correlation back to blocking workflow step. |
| Knowledge loop | `knowledge_asset.proposed`, `knowledge_asset.approved`, `persona.updated`, `workflow_template.updated` | Encodes continuous improvement. |
| Legacy bridge | `legacy_observation.captured`, `legacy_requirement.extracted`, `legacy_action.completed` | Adapter-specific payloads must remain isolated from core events. |
| Telemetry | `metric.recorded`, `quality_issue.detected`, `learning_pattern.detected` | May be aggregated for analytics. |

## API and Interface Requirements

No concrete protocol is selected yet. Any future API design must support:

- Workflow template CRUD and version promotion.
- Workflow run creation, state inspection, cancellation, pause, and resume.
- Task assignment, artifact submission, completion signaling, and recovery.
- Persona definition CRUD, lineage inspection, and version promotion.
- Knowledge asset query, proposal, review, and publication.
- Approval queue operations and decision submission.
- Event subscription, replay/audit access, and correlation lookup.
- Legacy adapter registration, health, permission, and run-scoped access.

## Non-Functional Requirements

### Reliability and Operability

- Workflow state transitions must be durable and reconstructable from audit or
  event history.
- Repeated agent failures, token-budget issues, and blocked approvals must be
  visible as operational signals, not buried in logs.
- Event consumers must be idempotent.
- Long-running workflows must be resumable after runtime interruption.

### Observability

- Every workflow run, task, approval, event, persona version, and knowledge-loop
  update must have a correlation identifier.
- Metrics should include workflow completion rate, step duration, approval cycle
  time, agent partial-completion rate, exception class, and knowledge-loop
  adoption.
- Logs and traces should distinguish platform behavior from adapter behavior.

### Security and Governance

- Persona permissions, tool access, and legacy adapter access must be explicit.
- Human approval records must preserve decision rationale and evidence reviewed.
- Knowledge promotion must be reviewable when it affects future agent behavior.
- Sensitive data handling, tenancy isolation, compliance controls, and retention
  requirements are open decisions.

### Extensibility

- New workflow domains should be added by configuration/templates/adapters
  rather than modifying core runtime behavior.
- Legacy bridge adapters should be independently deployable or replaceable if
  the eventual stack supports that separation.
- Event contracts must allow domain-specific extensions without changing core
  event semantics.

## Assumptions

- The current orchestration framework and playbook provide valid operating
  patterns for task cards, role agents, completion signals, deterministic
  validation, and feedback loops.
- Synapse will need both autonomous agent execution and human review, not one or
  the other exclusively.
- SME grounding depends on curated canonical assets and review workflows; raw
  notes alone are not sufficient runtime truth.
- Legacy bridge functionality is part of the architecture thesis, but MVP legacy
  systems and adapter priorities are not yet known.

## Open Decisions

- Concrete workflow runtime and state persistence mechanism.
- Concrete event transport, event schema format, and schema registry approach.
- API protocol choices and external integration authentication model.
- Data storage choices for workflow state, audit records, knowledge assets,
  persona definitions, and telemetry.
- Tenancy model and access-control model.
- Human approval policy model and default risk thresholds.
- Knowledge confidence scoring, expiration, and SME validation process.
- Visual workflow designer data format and template promotion workflow.
